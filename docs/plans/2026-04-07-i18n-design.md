# Internationalization of ClawForge

**Date:** 2026-04-07
**Status:** Approved

## Problem

All agent instruction files (SOUL.md, AGENTS.md, IDENTITY.md, SKILL.md) and
LLM prompts in orchestration.py are written in Russian. This limits the
framework to Russian-speaking users and reduces LLM instruction-following
quality (models perform better with English instructions).

## Goals

1. All framework files and LLM prompts in English (better LLM compliance,
   international accessibility)
2. User-facing language configurable at install time (en/ru/es)
3. Language changeable at any time via Telegram chat
4. Created agents also use English instructions with a language directive
   for user communication

## Solution

Hybrid approach:
- **Long texts** (SOUL.md, prompts): written in English, with a single
  `{{LANGUAGE_INSTRUCTION}}` placeholder for user-facing agents
- **Short strings** (notifications, errors): language pack (`i18n.py`)
  with translations for 3 languages

## Config

### `.clawforge_config` (replaces `.telegram_id`)

```json
{
  "telegram_id": "541534272",
  "language": "ru"
}
```

Supported values: `"en"`, `"ru"`, `"es"`.

### How telegram_id is determined

Auto-detected from OpenClaw pairing data at
`~/.openclaw/credentials/telegram-default-allowFrom.json` during `setup.py`
install. User never enters it manually.

### Language selection

- At install: `setup.py` prompts `Select language (en/ru/es) [en]:`
- At runtime: user writes to Architect in Telegram ("change language to
  Spanish") → Architect calls `main.py config --language es` →
  recommends `/new` to apply

### Migration from `.telegram_id`

`setup.py --update` detects old `.telegram_id`, creates `.clawforge_config`
with `language: "en"` (default), removes `.telegram_id`.

## Agent files — all English

### Pipeline agents (analyst, developer, reviewer, tester)

Translated to English entirely. No language placeholders — they never
communicate with users. They receive prompts and return JSON.

### Architect

Translated to English + one placeholder in SOUL.md:

```markdown
## Communication style

{{LANGUAGE_INSTRUCTION}}
```

Substituted by `setup.py` at install/update:
- `en` → `Always communicate in English.`
- `ru` → `Always communicate in Russian (на русском языке).`
- `es` → `Always communicate in Spanish (en español).`

Welcome message: written in English in the template. Architect translates
it naturally following the language instruction. No separate welcome
templates per language.

### Created agents

Developer generates all agent files in English (instruction in Developer
SOUL.md: "Write all agent files in English").

In each created agent's SOUL.md, Developer adds:
```
Always respond to the user in {language}.
```

Developer learns the user language from the developer_prompt:
```
User language: Russian. Add instruction "Always respond to the user
in Russian" to the agent's SOUL.md.
```

## Language pack (`src/i18n.py`)

Dict-of-dicts with ~25-30 user-facing strings:

```python
STRINGS = {
    "agent_created": {
        "en": "Agent '{name}' created and ready.",
        "ru": "Агент '{name}' создан и готов к работе.",
        "es": "Agente '{name}' creado y listo."
    },
    "agent_extended": {
        "en": "Agent '{name}' updated.",
        "ru": "Агент '{name}' обновлён.",
        "es": "Agente '{name}' actualizado."
    },
    ...
}

_language = "en"

def init(language):
    global _language
    _language = language

def t(key, **kwargs):
    entry = STRINGS.get(key, {})
    text = entry.get(_language, entry.get("en", key))
    return text.format(**kwargs) if kwargs else text
```

### What goes into language pack

- Telegram notifications via `send_notification()` (pipeline results,
  bind, delete, rollback, cancel)
- Error messages sent to user in Telegram
- `format_notification()` output in orchestration.py

### What does NOT go into language pack

- CLI print() for server admin — stays in English
- LLM prompts (analyst, developer, reviewer, tester) — English, not translated
- Agent instruction files (SOUL.md) — English, not translated
- Comments in code — English

## Language change flow

`main.py config --language es`:
1. Updates `.clawforge_config` with new language
2. Updates Architect SOUL.md — replaces `{{LANGUAGE_INSTRUCTION}}`
   with new language directive
3. Updates all created agents — finds and replaces language instruction
   line in their SOUL.md (`"Always respond ... in X"` → `"... in Y"`)
4. Prints confirmation

Architect recommends `/new` after language change so new session picks
up updated SOUL.md.

Pipeline agents (analyst, developer, reviewer, tester) are NOT updated
on language change — they always work in English.

## Backward compatibility

- Missing `.clawforge_config` → fallback to `.telegram_id` for ID +
  `language: "en"` default
- Existing created agents without language instruction → work as before
  (respond in whatever language the user writes in); updated on next
  `config --language` or manual fix
- `setup.py --update` handles migration automatically

## What does NOT change

- `src/registry.py` — no changes
- Workspace structure — no changes
- Versioning/rollback — no changes (except strings via `t()`)
- Pipeline flow (analyst → developer → reviewer → tester) — no changes
- OpenClaw integration — no changes
- Per-bot architecture — no changes
