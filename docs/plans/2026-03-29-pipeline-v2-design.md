# ClawForge Pipeline v2 — Дизайн

## 1. Контекст и проблемы

### 1.1. Выявленные проблемы (на примере github_bazaar)

Агент github_bazaar прошёл через конвейер 6 раз (1 create + 5 extend). Каждый раз тестер и валидатор ставили `approved: true` без замечаний. Все реальные проблемы были обнаружены и исправлены пользователем вручную через архитектора:

| Проблема | Кто должен был поймать | Почему не поймал |
|---|---|---|
| Парсинг stars today не работал для ряда репо | Tester | Проверяет текст SOUL.md, не реальное поведение |
| Кэширование данных между сессиями | Tester/Validator | Нет шага "запустить агента и посмотреть" |
| 5 дублирующихся блоков в SOUL.md | Developer | При extend переписывает весь файл с нуля |
| Экранирование `\#` в хэштегах | Tester | Нет проверки Telegram-специфики |
| Playwright не установлен | Никто | Конвейер не умеет ставить зависимости |
| Скрипт обрезки скриншота | Никто | Конвейер не умеет генерировать скрипты |
| Heartbeat/cron не настроен | Analyst | Не связал "каждый день в 10:00" с needs_heartbeat |

### 1.2. Корневые причины

1. **Тестер и валидатор — rubber stamp.** Проверяют документацию, а не поведение. Как code review без тестов.
2. **Developer при extend раздувает артефакты.** Переписывает SOUL.md целиком, добавляя дубли.
3. **Валидатор дублирует тестера.** За всю историю отклонил агента 1 раз — и то по причине, найденной тестером.
4. **Нет механизма для скриптов и зависимостей.** Developer генерирует только текстовые артефакты.
5. **Нечёткие инструкции аналитику по heartbeat.** Аналитик не всегда связывает расписание в задаче с needs_heartbeat.

## 2. Архитектура Pipeline v2

### 2.1. Агенты конвейера: 4 вместо 5

| # | Агент | Роль | Статус |
|---|---|---|---|
| 1 | **Analyst** | Анализ задачи → требования + тест-кейсы | Усилен |
| 2 | **Developer** | Требования → артефакты (включая скрипты) | Усилен |
| 3 | **Reviewer** | Статическая проверка артефактов + деплой | Новый (замена Tester) |
| 4 | **Tester** | Реальный запуск агента, сверка с требованиями | Новый (замена Validator) |

**Validator удаляется.** Его проверки (качество, безопасность, готовность) поглощены Reviewer-ом.

### 2.2. Два пути работы

#### Путь 1: Полный конвейер

**Когда:** `create_new`, крупный `extend_existing` (новые capabilities, новые skills).

```
Analyst → Developer → Reviewer ←→ Developer (до 3 retry)
                                      ↓
                                   Deploy → Tester
                                              ↓ (если проблема)
                                   Developer → Reviewer → Deploy → Tester (до 2 retry)
```

Управляется кодом `orchestration.py`. Архитектор запускает конвейер и получает результат.

#### Путь 2: Быстрый фикс

**Когда:** точечные правки — текст, формат, стиль, фикс бага.

```
Архитектор правит сам → Deploy → Tester
                                    ↓ (если проблема)
                    Архитектор правит → Deploy → Tester (до 2 retry)
```

Управляется инструкциями архитектора (SOUL.md, AGENTS.md). После любой прямой правки агента — обязательный вызов тестера как субагента.

**Критерии выбора пути (для архитектора):**
- Новый агент → полный конвейер
- Новая capability / skill → полный конвейер
- Правка текста, формата, стиля, фикс бага → быстрый фикс

**Тестер при быстром фиксе:** аналитик не вызывается, поэтому архитектор сам формулирует тестовое сообщение и ожидаемое поведение при вызове тестера как субагента.

### 2.3. Retry-политика

| Фаза | Retries | Обоснование |
|---|---|---|
| Reviewer↔Developer (статика) | до 3 | Дёшево (LLM-вызовы без деплоя), структурные правки поддаются итеративному исправлению |
| Tester→Developer (рантайм) | до 2 | Дороже (деплой + реальный запуск), но 2 retry закрывает каскадные баги (фикс одного ломает другое) |

**Рантайм retry — поток данных:** при отклонении тестером developer получает: оригинальные артефакты + `agent_response` (реальный ответ агента) + `issues` (что не так). Developer делает точечный фикс артефактов. Reviewer повторно проверяет статически. Deploy обновляет файлы. Tester повторно тестирует.

**При исчерпании retry:** конвейер честно сообщает пользователю что не так, с конкретным описанием проблемы от тестера. Без бесконечных циклов.

**Сравнение с текущим пайплайном:**

| | Текущий (v1) | Новый (v2) |
|---|---|---|
| Max вызовов Developer | 8 | 6 |
| Max вызовов LLM конвейера | ~14 | ~14 |
| Max реальных запусков агента | 0 | 3 |
| Что ловят retries | Формальные несоответствия (редко срабатывает) | Дубли, платформа (reviewer) + реальные баги (tester) |
| Типичный happy path | 5 вызовов, 0 реальных тестов | 5 вызовов + 1 реальный тест |

## 3. Детальные изменения в агентах

### 3.1. Analyst

**Новое — тест-кейсы.** Аналитик добавляет в JSON-ответ:
```json
{
  "test_message": "сгенерируй пост",
  "expected_behavior": "пост содержит 5 репо с названием и ссылкой, у каждого есть звёзды (число или Xk) и язык программирования, в конце 2-3 хэштега без экранирования"
}
```

Тест-кейс формулируется на основе requirements — аналитик лучше всех понимает что должен делать агент.

**Новое — чёткое правило heartbeat.** Инструкция аналитику:
> Если в задаче есть ЛЮБОЕ указание на расписание ("каждый день", "раз в час", "в 10:00", "ежедневно", "по утрам") — ОБЯЗАТЕЛЬНО ставь `needs_heartbeat: true` и заполняй `heartbeat_schedule` (cron-выражение) и `heartbeat_message`. Без исключений.

**Убрано:** `agent_type: "skill"` из списка допустимых значений. Скилл создаётся через `extend_existing`.

**Остаётся:** `decision: "automation_only"` — для cron-задач без своего агента.

### 3.2. Developer

**Новое — поле scripts.** Формат ответа расширен:
```json
{
  "soul_md": "...",
  "agents_md": "...",
  "identity_md": "...",
  "skills": {"skill-name": "SKILL.md content"},
  "data_files": {"state.json": "{}"},
  "scripts": {"screenshot_trending.js": "const { chromium } = require('playwright')..."},
  "system_deps": ["playwright"]
}
```

- `scripts` — исполняемые файлы, сохраняются в `workspace/<agent>/scripts/`
- `system_deps` — системные зависимости, устанавливаются при деплое

**Новое — анти-дубли при extend.** Инструкция developer-у:
> При extend_existing ты получаешь текущий SOUL.md агента. ОБНОВИ существующий файл — не переписывай с нуля. Правила:
> - Если инструкция уже есть — не добавляй её повторно
> - Если нужно изменить существующее правило — замени его, не дублируй
> - Следи за объёмом: если SOUL.md вырос более чем на 30% — проверь нет ли повторов
> - Один факт = одно место в документе

### 3.3. Reviewer (новый, заменяет старый Tester)

**Статическая проверка артефактов.** Все проверки, включая поглощённые от валидатора:

1. SOUL.md описывает все capabilities из требований
2. Skills покрывают все функциональные потребности
3. Нет противоречий в инструкциях
4. AGENTS.md содержит правила workspace
5. IDENTITY.md содержит имя, описание, эмодзи
6. Агент автономен — нет команд /main, /set, switch
7. Общее качество — понятна ли роль, готов ли к пользователю
8. Безопасность — нет деструктивных действий

**Новые проверки:**

9. **Дубли в SOUL.md** — нет повторяющихся блоков с одинаковым смыслом (особенно при extend)
10. **Платформо-специфика** — для Telegram: не экранировать `#`, правильный `parse_mode`, корректный формат Markdown
11. **Раздувание при extend** — если объём SOUL.md вырос значительно, проверить обоснованность
12. **YAGNI** — нет лишних возможностей за рамками требований

**После одобрения — деплоит** артефакты (вызов deploy).

Формат ответа:
```json
{
  "approved": true/false,
  "issues": ["список проблем"],
  "fixes": ["предложения по исправлению"]
}
```

### 3.4. Tester (новый, заменяет старый Validator)

**Реальный запуск агента.** Алгоритм:

1. Получить `test_message` и `expected_behavior` из требований аналитика
2. Отправить `test_message` задеплоенному агенту: `openclaw agent --agent <name> --message <test_message> --timeout 600`
3. Получить реальный ответ агента
4. Сверить ответ с `expected_behavior`:
   - Содержит ли ответ ожидаемые элементы?
   - Нет ли ошибок в формате?
   - Нет ли `н/д` где должны быть данные?
   - Соответствует ли стиль/язык требованиям?
5. Вернуть результат

Формат ответа:
```json
{
  "approved": true/false,
  "agent_response": "первые 500 символов ответа агента",
  "issues": ["конкретные проблемы в поведении"],
  "reason": "общая оценка"
}
```

**Работает и для create, и для extend** — агент уже задеплоен после reviewer-а.

**Тестовый запуск — реальный.** Тестер отправляет реальное сообщение живому агенту. Если агент использует внешние API (GitHub, Telegram Bot API и т.п.) — тестовый запуск реально к ним обращается. Это намеренно: мы проверяем именно реальное поведение, не мок.

**Тест-кейс при extend.** Аналитик генерирует test_message для проверки нового функционала. Если нужно проверить что старый функционал не сломан — аналитик может включить это в expected_behavior (например: "пост содержит 5 репо И хэштеги И скриншот").

### 3.5. Architect

**Новое — два пути в инструкциях.** Добавить в SOUL.md / AGENTS.md:

> При запросе на доработку агента оцени масштаб:
> - Новая capability, новый skill, значительное изменение поведения → запусти полный конвейер через skill claw-forge
> - Точечная правка текста, формата, стиля, фикс конкретного бага → исправь сам:
>   1. Прочитай текущий SOUL.md / SKILL.md агента
>   2. Внеси точечные изменения
>   3. Запиши обновлённый файл
>   4. ОБЯЗАТЕЛЬНО вызови тестера как субагента для проверки результата

### 3.6. Уведомления пользователя

Уведомления формируются в коде `orchestration.py` на основе данных от тестера — не через LLM, не через архитектора. Архитектор не участвует после запуска конвейера, его контекст не тратится.

Принцип: **что сделано + что протестировано + следующий шаг** (если есть). Для ошибок — конкретная проблема.

Реализация: шаблоны в коде `orchestration.py` с подстановкой реальных данных из ответов аналитика (`test_message`, `description`) и тестера (`approved`, `reason`, `issues`). Примеры ниже:

**Новый агент создан:**
```
Агент 'github_bazaar' создан и готов к работе.
Тест пройден: отправил "сгенерируй пост" — агент вернул пост с 5 репо, звёздами и хэштегами.
Если есть токен Telegram-бота — пришли его чтобы привязать.
```

**Агент расширен:**
```
Агент 'github_bazaar' обновлён.
Добавлено: скриншот страницы trending перед публикацией.
Тест пройден: агент корректно генерирует пост со скриншотом.
```

**Автоматизация создана:**
```
Автоматизация 'github_bazaar-heartbeat' создана.
Расписание: ежедневно в 10:00 UTC.
```

**Конвейер не справился (retry исчерпаны):**
```
Не удалось создать агента после 2 попыток исправления.
Проблема: агент возвращает "н/д" вместо числа звёзд для репозиториев без поля "stars today".
Можешь уточнить задачу или попросить доработать.
```

**Переиспользование:**
```
Для этой задачи подходит существующий агент 'thesis_maker'.
```

Данные для формирования берутся из ответа тестера (`agent_response`, `issues`, `reason`) и аналитика (`description`, `test_message`).

### 3.7. Команды

Без изменений: `/list`, `/rm`. Создание и доработка — через естественный язык.

## 4. Расписание и cron

### 4.1. Единый механизм: OpenClaw cron

Только OpenClaw cron через `jobs.json` с `kind: "cron"`. Не системный crontab.

- `kind: "cron"` запускает агента строго по расписанию — один вызов, без лишних токенов
- Управляется системой (enabled/disabled, состояние, gateway restart)
- `add_heartbeat()` уже использует этот формат

### 4.2. Decision: automation_only

Остаётся для cron-задач без своего агента (например автоматизация на архитекторе). При `automation_only` создаётся только запись в jobs.json, без workspace и артефактов.
