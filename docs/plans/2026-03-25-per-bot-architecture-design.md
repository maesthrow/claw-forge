# Per-Bot Architecture — Each Agent Gets Its Own Telegram Bot

**Date:** 2026-03-25
**Status:** Approved
**Supersedes:** 2026-03-24-reliable-agent-switching-design.md

## Problem

The agent switching mechanism (bindings, /set, /main) is fundamentally broken:
- OpenClaw's `/new` and `/reset` resolve agents by most recent session, not by binding
- Race conditions between binding changes and native commands
- Complex code with multiple failure modes (switch_agent, greet_via_channel, _touch_telegram_session)

No amount of patching fixes the root cause: one Telegram bot shared between agents with dynamic routing.

## Solution

**Eliminate switching entirely.** Each agent gets its own Telegram bot. No shared routing, no bindings manipulation, no session confusion.

OpenClaw natively supports this via `channels.telegram.accounts` — multiple bot tokens, each bound to a specific agent permanently.

## Architecture

### Agent types and communication

**Main (architect):**
- Own Telegram bot (existing)
- Commands: `/list`, `/rm`, agent creation
- Can delegate tasks to other agents via native OpenClaw mechanism ("Передай задачу thesis_maker: ...")
- No `/set`, no `/main`

**Pipeline helpers (analyst, developer, tester, validator):**
- Internal agents, called programmatically via `call_agent()`
- No Telegram bot, no user interaction
- Return structured JSON responses

**Created agents (thesis_maker, linkedin_writer, ...):**
- Each gets own Telegram bot (token from user via BotFather)
- Fully autonomous, no switching, no commands beyond native OpenClaw (/new, /reset)
- User interacts directly through the agent's bot

### Bot binding flow

**Variant A — Token at creation time:**
```
User: "создай thesis_maker, вот токен: 123:ABC"
→ Pipeline creates agent
→ bind_agent_to_bot("thesis_maker", "123:ABC", user_id)
→ "Готово! Пиши @thesis_maker_bot"
```

**Variant C — Token later:**
```
User: "создай thesis_maker"
→ Pipeline creates agent
→ "Агент готов! Создай бота в @BotFather и пришли токен"
User: "вот токен для thesis_maker: 123:ABC"
→ bind_agent_to_bot(...)
→ "Привязано! Пиши @thesis_maker_bot"
```

### OpenClaw config after binding

```json
{
  "channels": {
    "telegram": {
      "accounts": {
        "default": { "botToken": "main-bot-token", "dmPolicy": "pairing" },
        "thesis_maker": { "botToken": "123:ABC", "dmPolicy": "pairing" }
      }
    }
  },
  "bindings": [
    { "agentId": "main", "match": { "channel": "telegram", "accountId": "default" } },
    { "agentId": "thesis_maker", "match": { "channel": "telegram", "accountId": "thesis_maker" } }
  ]
}
```

Bindings are **static** — set once at bot creation, never changed.

## Delegation (native OpenClaw)

Architect can delegate tasks to other agents by saying "Передай задачу агенту X: ..." — this is a **built-in OpenClaw feature**, no custom code needed. OpenClaw sees the agent in `agents.list` and routes the task.

This must be documented in architect's SOUL.md so it knows how to use this pattern. Example scenarios:
- User: "скорми thesis_maker этот текст: ..."
- Architect delegates to thesis_maker via native mechanism, returns result to user

## Token handling edge cases

When user provides a bot token along with a creation request, but the pipeline decides `reuse_existing` or `extend_existing`:
- The token is **not used** — no new agent was created
- Architect explains the result to the user naturally ("Для этой задачи подходит существующий агент X, новый бот не нужен")
- This is behavioral (LLM), not code — architect understands context from pipeline result

## Risk Assessment

- **Low risk:** uses documented OpenClaw multi-account feature
- **Breaking change:** existing agents (thesis_maker, linkedin_writer) will need bot tokens to be accessible via Telegram
- **Migration:** existing agents keep working via architect delegation; Telegram access added when user provides bot token
- **Simplification:** removes ~150 lines of switching code, eliminates entire class of bugs
