# ClawForge — System Architecture

## What It Is

ClawForge is a layer on top of OpenClaw that automates AI agent creation through a pipeline of four specialized agents. The user describes a task to the main agent (architect) in Telegram, and the pipeline designs, develops, tests, and deploys a new agent into OpenClaw.

Each created agent is a full-fledged OpenClaw agent with its own workspace, sessions, and Telegram bot.

## Key Concepts

### Per-Bot Architecture

Each agent is bound to **its own** Telegram bot. There is no dynamic agent switching, no shared routing. Architect lives in its own bot, thesis_maker in its own, avatar_prompt in its own.

Implemented via a native OpenClaw feature — `channels.telegram.accounts`:

```json
{
  "channels": {
    "telegram": {
      "accounts": {
        "default": { "botToken": "architect-token" },
        "thesis_maker": { "botToken": "thesis-maker-token" }
      }
    }
  },
  "bindings": [
    { "agentId": "main", "match": { "channel": "telegram", "accountId": "default" } },
    { "agentId": "thesis_maker", "match": { "channel": "telegram", "accountId": "thesis_maker" } }
  ]
}
```

Bindings are static — set once during bot binding and never changed.

### Agent Creation Pipeline

When the user asks to create an agent, architect launches a pipeline of 4 internal agents:

```
User → Architect → Pipeline:
  1. Analyst    — analyzes the task, checks existing agents, selects a strategy
  2. Developer  — generates SOUL.md, AGENTS.md, IDENTITY.md, skills
  3. Tester     — validates artifacts against requirements
  4. Validator  — final audit before deployment
  → Deploy     — creates workspace, registers in OpenClaw
```

Analyst selects one of the strategies:
- **create_new** — a new agent from scratch
- **extend_existing** — any modification to an existing agent: updating behavior, adding skills, changing schedule, format, etc. Developer receives the agent's current SOUL.md for updating
- **reuse_existing** — a suitable agent already exists, nothing to create
- **automation_only** — a cron task without an agent (heartbeat)

### Task Delegation

Architect can delegate tasks to created agents via OpenClaw's native sub-agent mechanism (sessions_spawn). For example: the user asks to generate an avatar prompt → architect finds avatar_prompt in the agent list → launches it as a sub-agent → returns the result.

### Telegram Bot Binding

Two scenarios:
- **During creation**: the user provides a token along with the creation request
- **After creation**: the agent is created, the user sends a token later

Binding: `main.py bind --agent <name> --token <bot_token>` → writes account + binding to `openclaw.json` → gateway hot-reload picks it up.

## Agent Types

### Architect (main)

The main agent, entry point. Workspace: `~/.openclaw/workspace`

- Communicates with the user via its own Telegram bot
- Manages agents: creation, deletion, bot binding
- Delegates tasks to specialized agents
- Commands: `/list`, `/rm`, task description for creation
- Uses the `claw-forge` skill for CLI operations

Files:
- `agents/architect/SOUL.md` — role, delegation rules, communication style
- `agents/architect/AGENTS.md` — workspace rules, configuration protection
- `agents/architect/IDENTITY.md` — name "ClawForge", emoji
- `skills/claw-forge/SKILL.md` — CLI commands (/list, /rm, bind, create, cancel)

### Pipeline Agents (analyst, developer, tester, validator)

Internal pipeline agents. Called programmatically, do not communicate with the user.

- Workspace: `~/.openclaw/workspaces/<name>`
- Called via `deploy.call_agent()` → `openclaw agent --agent <name> --message "..."`
- Respond strictly in JSON
- Do not use memory, do not write files

Files:
- `agents/<name>/SOUL.md` — role, response format
- `agents/<name>/AGENTS.md` — rules: JSON format, no memory, no questions

### Created Agents (thesis_maker, avatar_prompt, ...)

Fully autonomous agents. Each has its own Telegram bot.

- Workspace: `~/.openclaw/workspaces/<name>`
- All files are generated by the pipeline (developer)
- Unaware of architect, no switching, operate autonomously
- The user communicates directly through the agent's bot

Files (generated by developer):
- `SOUL.md` — role, expertise, style
- `AGENTS.md` — workspace rules
- `IDENTITY.md` — name, emoji, description
- `skills/` — task-specific skills

## Key Modules

### deploy.py

Interaction with OpenClaw via CLI. All operations are wrappers around `openclaw` commands.

| Function | What it does |
|---------|-----------|
| `create_agent_workspace()` | Creates a folder with SOUL.md, AGENTS.md, IDENTITY.md, skills, data_files |
| `register_agent()` | `openclaw agents add` + restores files over defaults |
| `update_agent_files()` | Updates files of an existing agent (SOUL.md, AGENTS.md, IDENTITY.md, skills, data_files) |
| `delete_agent()` | `openclaw agents delete` + unbind bot + cron cleanup + gateway restart |
| `add_heartbeat()` | Creates a cron job in jobs.json with native cron format (`kind: "cron"`, `expr`, `tz: "UTC"`) |
| `bind_agent_to_bot()` | Writes telegram account + binding to openclaw.json |
| `unbind_agent_bot()` | Removes account + binding from openclaw.json |
| `call_agent()` | `openclaw agent --agent <name> --message "..."` |
| `clear_pipeline_sessions()` | Clears pipeline agent sessions before launching the pipeline |
| `send_notification()` | `openclaw message send` to Telegram |
| `get_telegram_user_id()` | Reads from .telegram_id or env |

**Important note:** `openclaw agents add` overwrites workspace files with defaults. `register_agent()` saves our files to memory before the call and restores them afterward.

### main.py

CLI interface. Called by architect via the claw-forge skill.

| Command | Function |
|---------|---------|
| `create --task "..." --notify telegram:ID` | Launches the pipeline in the background (fork), result sent to Telegram |
| `cancel --notify telegram:ID` | Stop the pipeline + clean up partial artifacts |
| `list` | Shows agents from the registry (with sync from OpenClaw) |
| `search --query "..."` | Search by description and capabilities |
| `bind --agent <name> --token <token>` | Bind a Telegram bot |
| `delete --agent <name>` | Delete agent + unbind + cron cleanup + gateway restart |

**Important notes:**
- `bind` and `delete` modify openclaw.json, which triggers a gateway hot-reload. This interrupts delivery of architect's response to Telegram. Therefore, after the operation — `time.sleep(2)` + explicit `send_notification()`.
- `create` tracks the PID of the running pipeline in `logs/pipeline.pid`. Double-launch protection: if a pipeline is already running — refuse. `cancel` kills the process (SIGTERM + SIGKILL fallback) and cleans up partial artifacts.

### orchestration.py

Agent creation pipeline. The core of the system.

**Flow:**
1. `run_pipeline(task_description)` — entry point
2. Analyst receives the task + list of existing agents → returns JSON with decision
3. Based on decision: reuse → return, automation → create cron, create/extend → continue
4. Developer receives requirements → generates artifacts (SOUL.md, AGENTS.md, IDENTITY.md, skills, data_files). For extend_existing, also receives the agent's current SOUL.md. All static instructions are in developer's SOUL.md; the prompt contains only the analyst's JSON.
5. Tester validates artifacts → approved/rejected
6. If rejected → Developer fixes → Tester validates (up to 3 times)
7. Validator — final audit (1 retry)
8. Deploy: `create_new` → `create_agent_workspace()` + `register_agent()` + heartbeat + gateway restart. `extend_existing` → `update_agent_files()` + heartbeat update + gateway restart

**Cron tasks:**
- Use native OpenClaw format: `kind: "cron"`, `expr: "0 6 * * *"`, `tz: "UTC"`
- Any standard cron expressions are supported without conversion
- On extend, the heartbeat is recreated with the same name `{agent}-heartbeat` (idempotent)

**Resilience:**
- Pipeline agent sessions are cleared before each run (`clear_pipeline_sessions()`) — prevents context accumulation and rate limiting.
- API errors (rate limit, timeout) are separated from JSON errors. API retry: exponential backoff 5→15→45→135s (4 attempts). When exhausted — RuntimeError (does not proceed to useless JSON retry).
- JSON parsing: `parse_json_response()` tries: direct parse → boundary search → code block strip → fallback. `call_agent_with_retry()` retries with an explicit JSON instruction (up to 2 times).
- 2-second delay between pipeline steps to reduce API load.

### registry.py

SQLite storage for agent metadata. Supplements the native OpenClaw registry with fields: description, capabilities, type, timestamps.

| Function | What |
|---------|-----|
| `add_agent()` | INSERT, capabilities serialized to JSON |
| `get_agent()` / `list_agents()` / `search_agents()` | SELECT, capabilities deserialized automatically |
| `update_agent()` | Single UPDATE query (not a loop) |
| `sync_with_openclaw()` | Reconciles registry with `openclaw agents list --json`, removes discrepancies |

### setup.py

Installation/update/removal of ClawForge on the OpenClaw server.

- `install` — creates pipeline agents, configures architect, installs skill, protects files (chmod 444)
- `update` — updates SOUL.md/AGENTS.md/IDENTITY.md/SKILL.md for all agents
- `uninstall` — removes everything: agents, workspace, bindings, registry, logs

## OpenClaw Workspace Files

Each OpenClaw agent has a workspace — a folder with configuration files:

| File | Purpose | Managed by |
|------|-----------|---------------|
| SOUL.md | Role, expertise, behavior | ClawForge |
| AGENTS.md | Workspace rules, startup protocol | ClawForge |
| IDENTITY.md | Name, emoji, description | ClawForge |
| USER.md | User information | OpenClaw (default) |
| TOOLS.md | Local tools | OpenClaw (default) |
| HEARTBEAT.md | Heartbeat config | OpenClaw (default) |
| MEMORY.md | Long-term memory | OpenClaw (created automatically) |
| skills/ | Agent skills | ClawForge |

ClawForge overrides SOUL.md, AGENTS.md, IDENTITY.md. The remaining files are OpenClaw defaults and are not modified.

## Technical Decisions

### Why Per-Bot Instead of Switching

The first architecture used a single Telegram bot + dynamic switching via `openclaw agents bind`. Problem: `/new` and `/reset` in OpenClaw select the agent by **the last active session**, not by binding. Race conditions were irresolvable. Solution: each agent = its own bot, bindings are static.

### Why a Custom Registry (SQLite) When a Native One Exists

The native OpenClaw registry (`openclaw agents list`) stores only id, name, workspace path. Our registry adds: description, capabilities (for search), type, timestamps. `sync_with_openclaw()` on `/list` reconciles registries — if an agent was deleted from OpenClaw directly, it is also removed from ours.

### Why a Pipeline of 4 Agents Instead of One

Separation of concerns: analyst decides **what** to do (new/extend/reuse), developer decides **how** (generates files), tester checks **quality**, validator — final audit. Each is specialized and has a strict input/output format (JSON). Retry logic allows developer to fix errors based on tester feedback.

### Why Files Are Saved Before `openclaw agents add`

`openclaw agents add` creates a workspace and **overwrites** all files with defaults. Our SOUL.md, AGENTS.md, IDENTITY.md are lost. Therefore, `register_agent()` saves file contents to memory before the call and restores them afterward.

### Why sleep(2) + send_notification After bind/delete

Writing to `openclaw.json` (bind/unbind bot) triggers a gateway hot-reload. During reload, the Telegram connection is interrupted and architect's response is not delivered. A 2-second delay + explicit `send_notification()` is the workaround.

## Deployment

Server: Linux with OpenClaw installed. The project is cloned to `/opt/clawforge/`.

```bash
# Installation
cd /opt/clawforge
python3 setup.py

# Update (after git pull)
python3 setup.py --update

# Removal
python3 setup.py --uninstall
```

Deploying changes: `git push` → `ssh git pull` → `setup.py --update`.
