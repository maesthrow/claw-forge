# Улучшение пайплайна: полная загрузка контекста агентов

**Дата:** 2026-04-01
**Статус:** Утверждён

## Проблема

1. **extend_existing**: Developer при доработке агента видит только SOUL.md. AGENTS.md, IDENTITY.md, skills, scripts не загружаются — генерируются заново, что ломает или дублирует существующее.
2. **create_new + reference_agents**: поле `reference_agents` декоративное — Developer получает только имена, файлы референсных агентов не загружаются. Сценарий "создать нового по образцу существующего" не работает.
3. **Architect при доработке**: выбирает между быстрым фиксом и полным конвейером вслепую — не видя текущего состояния агента. Инструкция быстрого фикса ограничивает чтение только "SOUL.md / SKILL.md".

## Решение

### 1. Полная загрузка файлов при extend_existing

Новая функция `load_agent_files(workspace_path)` в `orchestration.py` — собирает все файлы агента:
- SOUL.md, AGENTS.md, IDENTITY.md
- skills/*/SKILL.md
- scripts/*

Формат подачи в промпт Developer-а:
```
Текущие файлы агента (ОБНОВИ их, не переписывай с нуля):

=== SOUL.md ===
<содержимое>

=== AGENTS.md ===
<содержимое>

=== IDENTITY.md ===
<содержимое>

=== skills/my-skill/SKILL.md ===
<содержимое>

=== scripts/fetch.js ===
<содержимое>
```

Переменная `current_soul_context` заменяется на `current_agent_context`.

Reviewer при extend тоже получает полный предыдущий контекст (не только SOUL.md).

### 2. Загрузка файлов референсных агентов при create_new

При `create_new` с непустым `reference_agents` — загрузить файлы каждого референса через `load_agent_files()`.

Формат подачи — отдельным блоком с другой обёрткой:
```
Файлы похожего агента для РЕФЕРЕНСА (используй как образец структуры и стиля,
но создавай новые файлы под новые требования, НЕ копируй):

--- Агент: thesis_maker ---

=== SOUL.md ===
<содержимое>

=== AGENTS.md ===
<содержимое>

=== skills/generate/SKILL.md ===
<содержимое>
```

### 3. Architect загружает контекст перед выбором пути

Изменение SOUL.md архитектора — секция "Доработка агентов":

```markdown
## Доработка агентов — выбор пути

При запросе на доработку существующего агента:

### Шаг 0: Загрузи контекст агента (ОБЯЗАТЕЛЬНО)
Прочитай все файлы агента — SOUL.md, AGENTS.md, IDENTITY.md, skills, scripts.
Пойми текущее состояние прежде чем что-то менять или запускать конвейер.

### Шаг 1: Оцени масштаб и выбери путь

**Полный конвейер (через skill claw-forge)**
Когда: новая capability, новый skill, значительное изменение поведения,
добавление скриптов.
→ Вызови skill claw-forge. В описании задачи кратко опиши текущее
состояние агента, чтобы Analyst получил контекст.

**Быстрый фикс (самостоятельно)**
Когда: точечная правка текста, формата, стиля, исправление конкретного бага.
1. Внеси точечные изменения в нужные файлы (не переписывай весь файл)
2. Запиши обновлённые файлы
3. ОБЯЗАТЕЛЬНО проверь результат:
   - Сбрось сессию агента
   - Запусти агента как субагента с тестовым сообщением
   - Если результат не тот — исправь и проверь снова (до 2 раз)
```

### 4. Обновление SOUL.md агентов пайплайна

**Analyst SOUL.md** — добавить правило:
> `reference_agents` — указывай только если существующий агент решает очень близкую задачу и его структура реально полезна как образец. Не указывай агентов которые просто "тоже работают с текстом" или "тоже используют API".

**Developer SOUL.md** — дополнить:
> При extend ты получаешь ВСЕ текущие файлы агента (SOUL.md, AGENTS.md, IDENTITY.md, skills, scripts). Обновляй каждый файл точечно. Если файл не требует изменений — верни его как есть. Не удаляй существующие skills и scripts если это не требуется явно.

> При create_new ты можешь получить файлы референсного агента — используй их как образец структуры и стиля, но создавай новые файлы под новые требования. Не копируй референс — вдохновляйся.

**Reviewer SOUL.md** — дополнить:
> При extend_existing: проверяй все файлы на раздувание и дубли, не только SOUL.md. Если skills или scripts не изменились — они должны остаться идентичными оригиналу.

**Architect AGENTS.md** — убрать упоминание "SOUL.md / SKILL.md" из секции быстрого фикса.

## Карта сценариев после изменений

| # | Сценарий | Decision | Что происходит |
|---|----------|----------|---------------|
| 1 | Новый агент с нуля | `create_new` | Analyst → Developer → Reviewer → Deploy → Tester |
| 2 | Новый агент по образцу существующего | `create_new` + `reference_agents` | То же + Developer получает файлы референса как образец |
| 3 | Доработка существующего (значительная) | `extend_existing` | То же + Developer получает **все** файлы агента для обновления |
| 4 | Доработка существующего (точечная) | Architect сам | Architect читает все файлы → правит → тестирует (до 2 retry) |
| 5 | Переиспользование как есть | `reuse_existing` | Architect сообщает: "такой агент уже есть" |
| 6 | Cron без нового агента | `automation_only` | Создаётся cron-задача для существующего агента |

## Что НЕ меняется

- Сценарии 1, 5, 6 — работают как есть
- Структура decision-типов Analyst — те же 4 варианта
- CLI (main.py), deploy (deploy.py), registry (registry.py) — без изменений
- Формат JSON ответов Analyst и Developer — без изменений
- Setup.py — без изменений

## Затрагиваемые файлы

- `src/orchestration.py` — новая функция `load_agent_files()`, изменение формирования промптов Developer и Reviewer
- `agents/architect/SOUL.md` — секция "Доработка агентов"
- `agents/architect/AGENTS.md` — секция быстрого фикса
- `agents/analyst/SOUL.md` — правило reference_agents
- `agents/developer/SOUL.md` — правила extend и reference
- `agents/reviewer/SOUL.md` — правило проверки всех файлов при extend
